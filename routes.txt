# Django routing in `contact360/docsai`

## How routing is structured

1. **Root config**  
   `docsai/docsai/urls.py` defines the main `urlpatterns` and `include()`s app URL modules. Each app is mounted at a prefix (e.g. `docs/`, `api/v1/`).

2. **App-level URLs**  
   Each app has its own `urls.py` with `path()` (and sometimes `re_path()`). Some apps also use **nested** `include()` (e.g. Durgasman’s `api/`) or **DRF** `DefaultRouter` (e.g. Tasks API).

3. **Namespaces**  
   Apps set `app_name = '...'`. The Documentation API v1 uses `namespace='api_v1'`. You reverse URLs like `documentation:page_detail` or `api_v1:pages_list`.

4. **DEBUG-only routes**  
   When `DEBUG=True`, `urlpatterns` is extended with `static()` for `MEDIA_URL` and `STATIC_URL`.

## Smaller tasks for “learn, understand, analyse”

1. **Map root → apps**  
   Trace each `include()` in `docsai/urls.py` to its app and list which URLs live under which prefix.

2. **Docs app deep-dive**  
   Split `documentation` routes into: dashboard, pages/endpoints/relationships CRUD, media, operations, legacy. Note which views are HTML vs JSON and how they use `dashboard_api`, `media_views`, etc.

3. **API layering**  
   Compare `api/v1` vs `api/v2` vs `docs/api/...`: same resources, different styles (REST vs unified views). List which consumers use which (e.g. frontend, Lambda, Postman).

4. **Conflicts and overlap**  
   Check `docs/<page_id>/` vs `docs/pages/<page_id>/`, `docs/operations/` vs `/operations/`, `/media/` vs `docs/media-manager/` and `docs/media/...`. Decide which are legacy and which to standardise.

5. **Orphan URL configs**  
   Confirm `codebase.api` and `operations.api` are unused; either wire them (e.g. `codebase/api/`, `api/operations/`) or remove/deprecate.

6. **Router-based APIs**  
   Inspect `TaskViewSet` and any other ViewSets: supported methods, `lookup_field`/`lookup_url_kwarg`, and corresponding routes from the router.

7. **Namespaces and reverse()**  
   List all `app_name` and `namespace` values. Grep for `reverse(` / `{% url %}` and ensure they use the correct namespace and name.

8. **Path converters**  
   List use of `<uuid:...>`, `<str:...>`, `<path:...>`, `<int:...>` and ensure views expect those types (e.g. `task_id` vs UUID).

9. **Static/media in DEBUG**  
   Review how `static()` is appended and when `MEDIA_URL` / `STATIC_URL` are used; ensure they don’t clash with app routes.

10. **Document and maintain**  
    Add a small `ROUTES.md` (or section in `ARCHITECTURE_OVERVIEW.md`) that lists prefix → app → main routes and any “legacy vs preferred” notes, and keep it updated when adding/changing URLs.

If you want, we can go through a specific app (e.g. `documentation` or `durgasflow`) and map each route to its view and template/response type next.
